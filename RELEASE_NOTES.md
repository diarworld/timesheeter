# Изменения 03.04.26

## Исправления

### 1. Исправлен повторный поиск задач Yandex Tracker

**Проблема:** При повторном поиске той же задачи в поле поиска отображалось "Нет данных", хотя запрос к API выполнялся корректно и возвращал результаты.

**Причина:** Сложная логика управления состоянием в хуке `useYandexIssuesSearchOptions`:

- Гонки состояний между `useMemo` и `useEffect` при синхронизации накопленных результатов поиска
- RTK Query возвращал закэшированные данные без повторного запроса
- Проблемы с таймингом debounce

**Решение:**

- Упрощён хук `useYandexIssuesSearchOptions` — теперь использует данные RTK Query напрямую без промежуточного состояния `accumulatedIssues`
- Добавлен параметр `refetchOnMountOrArgChange: true` для принудительного обновления данных при каждом монтировании компонента

**Затронутые файлы:**

- `src/entities/issue/yandex/lib/use-yandex-issues-search-options.ts`

# Изменения 13.03.26

## Исправления

### 1. Исправлен выбор задач при повторном открытии модального окна

**Проблема:** При первом открытии модального окна создания трека отображались назначенные пользователю задачи. Однако после поиска и отправки формы, а затем повторного открытия модального окна — список задач был пустым ("Нет данных").

**Причины:**

- Хук `useInitialValue` кэшировал ключ задачи навсегда, что приводило к некорректному поведению
- RTK Query возвращал закэшированные данные без обновления при повторном монтировании компонента
- В `userIssuesAsOptions` проверка данных была в неправильном порядке: сначала проверялся `userPageData` (только текущая страница), затем `accumulatedIssues` (все загруженные страницы)

**Решение:**

- Удалён хук `useInitialValue`, который постоянно кэшировал ключ задачи
- Добавлен параметр `refetchOnMountOrArgChange: true` в RTK Query для получения свежих данных при каждом открытии модального окна
- Изменён порядок проверки в `userIssuesAsOptions`: теперь сначала проверяются накопленные задачи (`accumulatedIssues`), затем данные текущей страницы
- Добавлен `destroyOnClose` в `TrackModalCreate` для полного сброса состояния при закрытии
- Исправлена очистка таймера в `useDebouncedState` при размонтировании компонента

### 2. Исправлены бесконечные рендеры в тестах

**Проблема:** Тесты падали с ошибкой "Too many re-renders" из-за использования `useMemo` с `setState`.

**Решение:** Обновлены моки в тестах: теперь возвращается `undefined` вместо пустых массивов, когда данных нет. Это предотвращает бесконечный цикл обновлений состояния.

### Улучшения

- Сохранена функциональность ленивой загрузки и пагинации
- Добавлена сортировка задач по приоритету (точные совпадения ключа показываются первыми)
- Улучшена обработка ошибок и edge cases
